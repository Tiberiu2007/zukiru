# ADR 0005 â€” Native windowing backends (X11 + Wayland), no GLFW

- **Status:** Accepted
- **Date:** 2026-07-04
- **Deciders:** claude-opus-4-8 (at the project owner's direction)
- **Supersedes:** the "pick GLFW" recommendation in
  [ADR 0003](0003-windowing-backend.md) (its *deferral* rationale still stands as

## Context

ADR 0003 shipped the `platform::Window` interface and deferred the concrete
backend, tentatively recommending **GLFW**. The project owner has since decided
**not to depend on GLFW** (or SDL): Zukiru will own its windowing with **native
per-OS backends**. The first targets are the two Linux display protocols â€”
**X11** and **Wayland** â€” with **Windows (Win32)** and **macOS (Cocoa)** left as
explicit future work.

The development environment has both stacks available: a live X11 server
(`DISPLAY=:0`) and the Wayland client/protocol/xkbcommon toolchain
(`wayland-client`, `wayland-protocols` + `wayland-scanner`, `xkbcommon`). There is
no running Wayland compositor here, so the Wayland backend is validated to build
and link but is runtime-tested only on a Wayland session.

## Decision

Implement windowing as **native backends inside `platform`**, selected at
runtime, behind the existing `Window` interface.

### Backends

- **X11** (`src/backend/x11/`) via **Xlib** â€” create/map a window, WM close
  protocol (`WM_DELETE_WINDOW`), and translate `XEvent`s (key, button, motion,
  scroll, configure/resize, focus) into `platform::WindowEvent`s. Keysyms are
  mapped to `platform::Key`.
- **Wayland** (`src/backend/wayland/`) via **libwayland-client** + **xdg-shell**
  (generated from `wayland-protocols` with `wayland-scanner`) + **xkbcommon** for
  the keyboard keymap. A `wl_shm` buffer is attached so the surface is visible
  before a GPU renderer exists. Pointer/keyboard come from `wl_seat`.

### Selection

`createWindow()` picks at runtime: **Wayland if `WAYLAND_DISPLAY` is set**,
otherwise **X11 if `DISPLAY` is set**, otherwise a clear `Error`. A backend that
is compiled out (its libraries weren't found) is simply skipped. Both are opt-out
via `ZUKIRU_WINDOW_X11` / `ZUKIRU_WINDOW_WAYLAND` CMake options (default ON when
the libraries are detected).

### Events & input

`platform` emits backend-neutral `WindowEvent`s and defines the **platform-level**
input enums (`Key`, `MouseButton`, `KeyMods`). It must not depend on the `input`
module (that would invert the layering). The `input` module (Layer 1) instead
gains a small **bridge** (`input/platform_bridge.hpp`) that pumps a window's
events into an `input::InputState`, translating `platform::Key` â†’ `input::Key`.
The two enums are defined with identical ordering so the bridge is a checked
`static_cast` (guarded by `static_assert`s on sentinels) rather than a large
switch; this makes `input` depend on `platform` (Layer 1 â†’ Layer 0, allowed).

### Build isolation

The Wayland protocol code generated by `wayland-scanner` is C and is compiled in
its own small static library **without** `zukiru::warnings`, so the engine's
`-Werror` warning set never polices generated third-party code.

## Consequences

- **No third-party windowing dependency** â€” full control, no vcpkg/GLFW; the cost
  is per-OS code and maintenance (accepted).
- Native handles are exposed (`nativeHandle()`/`nativeDisplay()`: `Window`+`Display`
  for X11, `wl_surface`+`wl_display` for Wayland) so the future Vulkan backend can
  build a surface without further plumbing.
- **Not headless-testable end-to-end**: opening a window needs a display, so the
  window-creation smoke test is hidden-tagged (`[.window]`, run on demand). The
  display-independent parts â€” backend availability, the input-bridge key mapping,
  event plumbing â€” are covered by normal unit tests.
- **Deferred:** Win32 and Cocoa backends; text/IME input (`WindowEvent` has no
  char event yet); gamepad input (a separate device backend for the `input`
  module); high-DPI/scale handling; multi-monitor/fullscreen mode switching.

## Alternatives rejected

- **GLFW / SDL** â€” rejected by the project owner; would be less code but adds a
  dependency and cedes control of the window/input path the engine wants to own.
- **XCB instead of Xlib** â€” lower-level and async-friendly, but Xlib is simpler to
  get correct first and is universally available; XCB can replace it later behind
  the same backend boundary if needed.
